# 暖禾智家全屋智能中控系统 (Nuano Smart Home)

[![OS](https://img.shields.io/badge/OS-OpenHarmony%20LiteOS--M-blue)](https://www.openharmony.cn/)
[![Agent](https://img.shields.io/badge/Agent-DeepSeek%20LLM-purple)](https://www.deepseek.com/)
[![License](https://img.shields.io/badge/License-China%20Software%20Cup-red)](#)

暖禾智家是一款深度基于 **OpenHarmony** 操作系统研发的全屋智能家居控制系统。系统采用云边端一体化架构，通过轻量级 MQTT 物联网协议实现高效稳定的设备互联，并深度集成 **DeepSeek 大语言模型** 构建专属智能代理。

> 💡 **核心设计规范：**
> *   **三层零信任安全底座：** 包含 TLS 传输层加密、AES-128-GCM 数据层载荷加密及 HUKS 硬件级隔离保护。
> *   **人机协同决策闭环：** 首创基于前端弹窗拦截与人工视觉复核的操作逻辑，完美阻断 AI 幻觉引发的物理误动作风险。
> *   **断网降级运行：** 具备高可靠边缘容错能力，外网不可用时自动退入纯传感器本地自动化模式。

## 🛠️ 1. 硬件连接规范

### 1.1 核心外设清单
1. **主控：** 普中-Hi3861 开发板（运行 OpenHarmony 轻量系统）。
2. **传感器：** HLK-LD2410C 毫米波雷达（人体微动检测）、DHT11 温湿度传感器。
3. **物理执行：** 三线制 LED 照明模块、SG90 模拟舵机（模拟门锁）、38kHz 红外发射模块（覆盖格力/美的/海尔三大品牌空调）。

### 1.2 物理接线总表
所有模块与开发板**必须共地（GND）**，建议供电适配器输出电流 **≥ 1A**，以防止舵机动作时瞬间欠压导致系统硬件复位。

| 边缘模块引脚 | 接 Hi3861 开发板引脚 | 电气说明 / 物理用途 |
| :--- | :--- | :--- |
| **LD2410C VCC / OUT** | 5V / `GPIO_11` | 必须 5V 供电，高电平代表有人存在 |
| **DHT11 VCC / DAT** | 3.3V / `GPIO_7` | 单总线数字气候环境采集 |
| **LED 模块 IN** | `GPIO_14` | 高电平点亮模拟室内照明 |
| **SG90 信号线 (橙色)** | `GPIO_5` | PWM 控制脉宽（0°关门 / 180°开门） |
| **红外发射 OUT** | `GPIO_8` | 38kHz 载波依次发射三品牌 Toggle 型空调电源码 |
| **板载 KEY1** | `GPIO_12` | 物理按键：用于切换“自动感应 ↔ 手动控制”模式 |

### 2.1 修改网络参数
编辑工程目录下 `demo/43_smart_home/template.c` 顶部的宏定义，配置您的 2.4GHz 无线局域网凭证与主题路径：

```c
/* WiFi 配置 —— 仅支持 2.4GHz 路由热点 */
#define WIFI_SSID       "您的局域网名称"
#define WIFI_PAWD       "您的局域网密码"

/* MQTT 配置 —— 默认使用 EMQX 国内公共服务器 */

#define MQTT_PORT       1883
#define MQTT_CLIENT_ID  "hi3861_smart_home"
#define TOPIC_CONTROL   "smart_home/control"  /* 设备订阅控制信道 */
#define TOPIC_STATUS    "smart_home/status"   /* 设备发布遥测信道 */


提示：在多人公共压测环境下，请在主题名后加入唯一标识后缀（如 smart_home/control_yicheng）以防串流干扰。

### 2.2 构建与刷写

1. **启用项目：** 编辑 `demo/BUILD.gn`，确保只取消 `43_smart_home:template` 的注释。


2. **环境全量编译：** 在 OpenHarmony 源码根目录下依次执行命令行：
```bash
hb set      # 选择对应产品线 pzkj / pz_hi3861
hb build -f # 全量编译，成功后输出 Hi3861_wifiiot_app_allinone.bin

```


3. **串口固件烧录：** 打开 Windows 端 **HiBurn** 工具，选择对应 COM 口，将波特率配置为 **921600**。勾选 `Auto burn` 并点击 `Connect`，手动按下开发板的 **复位键（RESET）** 触发刷写，进度条完成后断开连接，改用串口助手（波特率 115200）监视启动日志。

## ⚙️ 3. 零信任加解密数据流规范 (AES-128-GCM)

系统全链路业务载荷强制实施认证加密。数据在网络传输中统一采用大写的 **Hex 十六进制字符串** 格式：`Hex( AES-128-GCM密文 + 16字节Tag认证标签 )`。

### 3.1 远程原子指令速查

* `light_on` / `light_off` / `light_auto` ：强启照明 / 强关照明 / 恢复雷达自动托管。


* 注：系统底层的 DeviceManager 会自动将 AI 生成的结构化属性对象（如 `power:true`）无缝转换为上述对应的硬件原子化字符串指令投递至底层总线。




* `door_open` / `door_close` ：驱动门锁舵机至 180° / 0° 物理边界。


* `ac_on` / `ac_off` / `ac_model` / `ac_wind` ：红外广谱调控暖通空调属性。



### 3.2 下行控制密文生成 Python 示例

```python
from Crypto.Cipher import AES
from paho.mqtt import client as mqtt_client

KEY = bytes([0x2B,0x7E,0x15,0x16,0x28,0xAE,0xD2,0xA6,0xAB,0xF7,0x15,0x88,0x09,0xCF,0x4F,0x3C])
NONCE = bytes(range(12)) # 预置 96bit 随机数

cipher = AES.new(KEY, AES.MODE_GCM, nonce=NONCE)
ct, tag = cipher.encrypt_and_digest(b'{"cmd":"door_open"}')
hex_str = (ct + tag).hex().upper() # 密文在前，16字节 Tag 在后拼接转大写

c = mqtt_client.Client(mqtt_client.CallbackAPIVersion.VERSION1, "pc_ctrl_001")
c.connect("broker-cn.emqx.io", 1883)
c.publish("smart_home/control", hex_str)
c.disconnect()

```

## 🤖 4. AI智能管家协同控制闭环

系统的大模型控制链路严格遵循以下数据流动防线，将大模型随机产生的输出幻觉在边界层实施完美无损拦截：

```text
[1. 用户输入: 文字/语音自然语言]
       │
       ▼ (传递 Prompt)
[2. AI分析: 云端大模型融合温湿度解析推理]
       │
       ▼ (提取特征对象)
[3. 控制提议: 生成 Device/Action/Value 标准 JSON]
       │
       ▼ (触发后台线程挂起)
{4. 强制拦截: 前端弹出智能场景决策确认视窗}
       ├──► [判定偏差: 点击取消忽略] ──► [5. 策略栈静默销毁并降级闲聊]
       └──► [判定合理: 主动点击授权] ──► [6. DeviceManager: 标准化对象转化与分发]
                                                      │
                                                      ▼ (别名调用 HUKS TEE 密钥加密)
                                          ([7. 设备执行: MQTT 安全总线物理动作响应])

```

### 4.1 大模型强约束结构化输出规范

系统将 API 请求的随机度参数压缩至 **0.1**，强制要求大模型返回包含 `type`、`message` 与 `commands` 三个固定字段的标准复合 JSON 对象。当解析出明确的调温与灯光操作意图时，云端决策中枢生成如下格式的 JSON 数组分发至中控拦截层：

```json
[
  {
    "device": "ac",
    "action": "temperature",
    "value": 26
  },
  {
    "device": "light",
    "action": "power",
    "value": false
  }
]

```

### 4.2 AI智能微调原则与白名单边界

* 温度微调边界：针对“我有点热”、“体感偏冷”等模糊感官表达，系统强制要求模型基于当前传感器采集的真实温度标量进行±2℃至±3℃的动态步进微调，严禁输出超出正常物理区间的极端异常值。
* 白名单越权拦截：决策中枢仅放行空调与主灯的AI控制权限。任何企图操纵智能门锁等高敏感安防执行硬件的AI策略，均会在白名单合法性校验阶段被立刻捕获并静默丢弃，确保家庭隐私与物理边界安全。

* **零信任物理网络渗透测试：**
* 展示局域网内的Wireshark抓包数据，验证8883安全端口的公网长连接流量全量显示为TLS加密ApplicationData密文。

## 🔍 6. 快速缺陷排查诊断

**Q: 串口高频循环滚动打印[DHT11]检测失败，请检查接线!**
**A:** 固件上电自检阻塞。请核对单总线数据引脚物理上是否连通至GPIO_7。若电气导通后仍存在读数为0的报错，属于数据清洗层触发的零值过滤机制，请重新拔插电源。

**Q: 下发开门指令后，模拟舵机毫无动作，或者开发板突发死机重启复位？**
**A:** SG90模拟舵机在0°至180°旋转或发生外部机械卡死堵转时，瞬间抽取的尖峰电流极大。若直接用电脑USB或输出小于1A的劣质充电头供电，会导致总线瞬时欠压复位。请配置输出能力≥1A的高稳定度直流电源，并将模拟舵机的VCC进行独立外部供电（必须与开发板共地连接）。

**Q: 串口高频提示[AES]Tag校验失败!且硬件毫无响应**
**A:** 零信任总线校验未放行。请检查前端ArkTS应用通过HUKS密钥库别名调用的对称密钥属性，是否与边缘端硬编码的g_aes_key保持字节一致；另核对控制报文大写Hex转换尾部是否由于丢包残缺了16字节的验证标签。
